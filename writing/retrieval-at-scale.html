Retrieval at scale: what breaks when your OpenSearch index grows

A working draft — field notes on the failure modes that appear as an index goes from thousands to hundreds of millions of documents.

Small OpenSearch indexes are forgiving. You can ignore sharding, let mappings grow, paginate naively, and everything still returns in a few milliseconds. Scale changes that — and the failures rarely show up where you expect. They hide in the p99, in heap pressure, and in slowly rotting relevance. Here's the short list of what actually breaks, and the levers that fix each one.

TL;DR Size shards deliberately (tens of GB, not thousands of tiny ones). Keep mappings lean. Never deep-paginate — use search_after/PIT. Budget RAM for vector graphs before you turn on kNN. Separate indexing pressure from query latency. And measure relevance on a fixed eval set, because it drifts as the corpus grows.

1. Shards: too many small ones, or too few huge ones

Every shard is a Lucene index with fixed overhead — file handles, memory, and per-query coordination cost. Oversharding (a common default when you create indexes per day/tenant) means a single query fans out to hundreds of shards and pays coordination tax on every request. Undersharding means individual shards grow past the point where merges and recovery are painful.

  • Aim for shards in the tens of GB (a widely-used rule of thumb is ~10–50 GB), not hundreds of MB.
  • Watch total shard count per node — thousands of tiny shards is a heap and scheduling problem on its own.
  • Use rollover (ISM) so time-series indexes roll at a size/age target instead of accumulating.

2. Mapping explosion and field cardinality

Dynamic mapping is a trap at scale: unbounded keys (user-supplied JSON, event payloads) quietly create thousands of fields, and every field costs memory in the cluster state and slows mapping updates. High-cardinality keyword fields also inflate global ordinals and aggregation cost.

  • Turn off dynamic mapping (or set strict) on anything ingesting arbitrary JSON.
  • Model variable attributes as flattened fields instead of first-class ones.
  • Index only what you query; store the rest without indexing.

3. Deep pagination

from + size is fine for page 2 and catastrophic for page 2,000: each shard has to build and sort a result set of from + size before discarding almost all of it. At scale this is the classic OOM / latency spike.

Rule: if a user can page past a few hundred results, you don't want from/size — you want search_after with a Point-in-Time (PIT), or the scroll API for exports.

4. Vector/kNN memory is the quiet budget-killer

Turning on kNN (HNSW) changes your capacity math. The graph lives in memory, and its footprint scales with the number of vectors, their dimensionality, and the graph parameters (m, ef_construction). Teams enable semantic search on a whim and are surprised when nodes start swapping or off-heap memory blows past the container limit.

  • Estimate graph RAM before enabling it: roughly proportional to num_vectors × dim × 4 bytes × (1 + m-overhead).
  • Consider quantization (byte/PQ) to cut the footprint, and tune ef_search for the recall/latency point you actually need.
  • Segment merges rebuild graphs — expect CPU and memory spikes during heavy indexing.

5. Indexing pressure vs. query latency

The two workloads fight. A tiny refresh_interval makes documents searchable fast but multiplies segment churn and merge load, which shows up as query-latency tails. Heavy bulk indexing during peak read traffic is how a healthy cluster suddenly posts a bad p99.

  • Raise refresh_interval (or refresh on demand) for write-heavy pipelines that don't need second-fresh reads.
  • Separate hot indexing from serving where you can; throttle bulk during peak.
  • Track merge activity and segment count — they're leading indicators of latency trouble.

6. Hybrid search & rerank cost

Combining BM25 with vector similarity (hybrid) usually wins on relevance, but naive score fusion breaks because the two score distributions aren't comparable. And bolting a cross-encoder reranker on top adds real latency and cost per query.

  • Normalize before you fuse (min-max or RRF) — don't add raw BM25 and cosine scores.
  • Rerank only the top-k candidates, and set k from a measured latency budget, not a guess.
  • Track tokens (or ms) per resolved query as a first-class cost metric.

7. Relevance drifts as the corpus grows

The most insidious failure isn't an error — it's quality quietly degrading. Queries that returned the right doc at 1M documents return near-duplicates or stale content at 100M, and nobody notices until users complain.

  • Keep a fixed evaluation set of query → expected-result pairs and score it on every meaningful change.
  • Watch nDCG/recall@k over time, not just latency dashboards.
  • Re-tune analyzers, boosts, and rerank thresholds as the distribution shifts.

The through-line

Almost every one of these is the same lesson in a different costume: at scale, defaults optimize for the small case. The fix is to make the trade-offs explicit — a shard-size target, a mapping policy, a pagination contract, a memory budget for vectors, a latency budget for rerank, and a standing relevance eval. Instrument those early and the "mysterious" scaling failures mostly stop being mysterious.
