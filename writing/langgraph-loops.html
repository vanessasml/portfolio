LangGraph in production: taming loops and cyclic agent flows

A working draft — how to keep cyclic agent graphs useful without letting them spin, stall, or burn your budget.

The whole reason to reach for LangGraph over a linear chain is the cycle: an agent that can reflect, retry, call a tool, and loop back with new information. That loop is the feature. It's also exactly what turns a working demo into a 3am incident — an agent that never decides it's done, re-queries the same tool forever, or quietly triples your token bill. Here's what it takes to run cyclic graphs in production.

TL;DR Model loops as explicit nodes + conditional edges. Give every cycle a hard termination condition and a recursion_limit. Remember that every node and every subgraph adds latency on top of model time — parallelize and keep graphs shallow. Budget steps and tokens as first-class limits. Be deliberate about state reducers so message lists don't balloon. Checkpoint for resumability, and trace every transition — you can't debug a loop you can't see.

1. Make the loop explicit

A LangGraph app is a StateGraph of nodes connected by edges; a loop is just an edge that points back to an earlier node. Keep those cycles intentional and small. The common shape — agent → (tool? → agent) → end — is a conditional edge that routes back to the agent only while there's more tool work to do.

  • Route with a conditional edge function that inspects state and returns the next node (or END).
  • Keep the number of distinct cycle entry points small — one or two, not a web of back-edges.
  • If a subtask is really a DAG, don't force it into a loop; reserve cycles for genuine iterate-until-done work.

2. Every loop needs a way to stop

The number-one production failure is a loop with no reliable exit. Never rely on the model alone to decide it's finished — encode termination in the graph.

Belt and braces: a semantic "done" condition in your routing function, and a hard recursion_limit so the graph raises instead of looping forever.
  • Track an explicit iterations counter in state and stop at a ceiling.
  • Set recursion_limit on invocation as a safety net; handle the resulting error gracefully (return best-effort, don't crash the request).
  • Detect no-progress loops — same tool + same args as the previous step usually means "break."

3. Latency: every node — and every graph — adds up

This is the trap people underestimate. Each node is a step the runtime schedules, and the orchestration overhead is real and measurable — separate from model time. Practitioners have reported a LangGraph tool-calling run around ~15s where the equivalent direct model call was ~5s, and there's an open issue about nodes showing high latency unrelated to LLM execution (see also this discussion of framework overhead vs. a direct call). In a sequential graph this accumulates: N nodes is N hops of orchestration on top of N model calls.

The danger multiplies when you compose multiple graphs: a supervisor over several subgraphs, or pipelines that chain graph-of-graphs. Coordination cost — and the debugging surface — grows fast, and beyond ~5 agents it grows roughly exponentially.
  • Parallelize independent branches instead of chaining them, and run I/O-bound nodes async so tool calls overlap — the biggest lever on total latency. [scaling subgraphs & map-reduce]
  • Pre-warm the compiled graph — recompiling on every request adds ~70 ms of cold-start you don't need to pay.
  • Keep graphs shallow. Deeply nested conditional branches and highly interconnected nodes slow coordination; past ~5 agents, monitoring and debugging complexity explode. [multi-agent architecture analysis]
  • Trace per-node time to catch the "unobserved latency cascade" — orchestration overhead hides behind total latency until you measure each hop. [LangGraph in production: latency]

4. Budget cost and tokens like a limit, not a hope

Each trip around the loop is another model call (and often another tool call). Costs are multiplicative in the number of iterations, so a small loop that "usually runs twice" can occasionally run twenty times on a hard input.

  • Cap total tokens/steps per request and short-circuit when the budget is hit.
  • Keep the per-iteration prompt lean — don't resend the entire history every turn if you can summarize it.
  • Log tokens-per-completed-task, not just per-call, so runaway loops are visible in dashboards.

5. Be deliberate about state reducers

LangGraph merges each node's output into shared state via reducers. The classic footgun is the message list: with an append reducer it grows every iteration, inflating context and cost until you hit the window. Decide, per field, whether a node should append or overwrite.

  • Use append reducers for genuine history; overwrite for scratchpad/working fields.
  • Trim or summarize the message history inside the loop, not just at the end.
  • Keep large tool outputs out of the accumulating state — reference them, don't inline them every turn.

6. Checkpoint for resumability and human-in-the-loop

Long cyclic runs fail — a tool times out, a rate limit hits, a human needs to approve a step. A checkpointer persists state at each super-step so you can resume instead of restarting, and it's what makes interrupt-and-approve workflows possible.

  • Use a durable checkpointer (not in-memory) in production; key it by thread/session.
  • Design explicit interrupt points for approvals rather than blocking inside a node.
  • Make nodes idempotent where you can, so a resumed step doesn't double-execute side effects.

7. You can't debug a loop you can't see

Cyclic behavior is opaque without tracing. Instrument every node transition so you can answer "why did it loop again?" after the fact.

  • Emit a trace per super-step: node, routing decision, tokens, and a state diff.
  • Alert on iteration count and per-request cost crossing thresholds.
  • Capture the routing function's inputs — most "mystery loops" are a router returning the wrong branch.

8. Test the cycles, not just the happy path

Unit-test routing functions directly with crafted states, and run the whole graph against deterministic (stubbed) tools so a loop's behavior is reproducible.

  • Assert termination: every test graph must reach END within the step budget.
  • Add adversarial inputs that tempt the agent to loop (ambiguous tasks, flaky tools).
  • Snapshot-test state after each step to catch reducer regressions.

The through-line

Loops are what make LangGraph worth using — and the discipline is to treat termination, budget, and observability as first-class parts of the graph, not afterthoughts. Design the exit conditions before the happy path, cap the cost, watch every transition, and the same cycles that cause 3am pages become the thing that makes your agents genuinely capable.
